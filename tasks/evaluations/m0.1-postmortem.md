## 3. Error Analysis

### Timeouts (5 total)

| Time | Agent | Type | Impact |
|---|---|---|---|
| 14:34:44 | CEO | timeout waiting for agent_end | Run lost, Paperclip retried |
| 14:36:03 | Researcher | timeout waiting for agent_end | Run lost, retried |
| 14:40:08 | Writer | timeout waiting for agent_end | Run lost, retried |
| 14:45:36 | CEO | timeout waiting for agent_end | Run lost |
| 14:45:57 | CEO | timeout waiting for agent_end | Run lost |

Pattern: Pi process starts successfully (pi_ready fires), then hangs during execution. No stderr output. Likely cause is MiniMax API hanging during inference on long tool-use chains with no streaming heartbeat. All 5 timeouts follow the same signature: successful start, no error, silent hang until BRIDGE_TIMEOUT_MS expires.

### Stale context escalations (3 total)

- EVA-57: "SYSTEMIC PROBLEM: Agent adapter failures blocking all research work" -- stale context from crash loop era. CEO saw error history from prior failed runs in Paperclip issue comments and believed agents were still broken.
- EVA-58: Similar stale context escalation referencing EVA-28 (a crash-loop era issue).
- EVA-69: "CEO Escalation: Researcher Adapter System Failure" -- created at 14:56, four minutes after EVA-55 was already marked complete. CEO saw old error state in a post-completion wake.

These are not real problems. They reflect stale Paperclip issue comments from the crash loop persisting into the clean run's context window. The CEO correctly identified failure patterns but could not distinguish historical failures from current state. This is a context hygiene problem, not a reasoning problem.

## 4. Back-and-forth Analysis

### Invocation counts (approximate from logs)

- CEO: ~15-20 invocations across heartbeats, assignments, and continuations
- Researcher: ~8-10 invocations
- Writer: ~6-8 invocations
- Data: ~4-5 invocations

### Unnecessary round-trips observed

1. **Heartbeat wakes during active work.** Bridge serialization means heartbeats queue behind active work, but Pi still processes the "check your inbox" prompt even when already working on the issue. Wastes 1-2 turns per heartbeat.

2. **CEO self-assigned research tasks (EVA-62, 66).** Should have delegated to Researcher or Data. This created 2 extra CEO runs that could have been parallel on other agents, and violated the CEO coordination-only constraint.

3. **Duplicate write tasks.** EVA-56, EVA-67, and EVA-68 all overlap as "write the report." CEO created 3 writing tasks instead of 1. EVA-67 was never completed. The Writer finished EVA-56 while CEO separately completed EVA-68.

4. **Post-completion wakes.** After EVA-55 was marked done at 14:52, CEO still received wakes and created EVA-69 escalation at 14:56. Agents should not process wakes for completed parent issues.

## 7. Workspace Analysis

### CEO workspace

Contains triage audit JSONL files per issue (`/workspace/{issue_id}/triage/audit.jsonl`) -- this is the triage-workflow extension logging phase transitions. Also has 2 stale artifacts from prior runs and a report.md.

### Researcher workspace

Two issue directories with structured workproduct findings:

- `/workspace/e445a2c3-.../workproduct/findings/` -- 6 finding JSON files for one session
- `/workspace/41f724e7-.../workproduct/findings/` -- 10+ finding JSON files for another session

Findings are LOCAL only -- not published to the artifact service. The researcher publishes the aggregate research document via write_artifact, but individual findings stay on the workspace volume.

### Data workspace

Minimal: 2 markdown files in one issue directory. No workproduct findings structure. Data agent wrote directly to workspace and published via write_artifact.

### Writer workspace

Rich structure with manifest.json, skeleton.json, and individual section files per issue. Multiple issue directories showing different report writing sessions. Writer correctly follows the skeleton-section-stitch-polish pipeline from the writing-style extension.

### Key finding

Workproduct findings (structured JSON with ADMIRALTY grading) exist only in researcher workspace volumes. They are NOT in the artifact store. If the workspace volume is destroyed (`docker compose down -v`), these findings are lost. The published artifacts are aggregate markdown documents, not the structured findings themselves. This is the highest-severity data loss risk identified in this evaluation.

## 8. Recommendations

Prioritized by severity.

### P0 -- Data loss risk

1. **Publish structured findings to artifact service, not just aggregate documents.** Individual findings with ADMIRALTY grades are more valuable than prose summaries. They are queryable, composable, and auditable. Currently they exist only on ephemeral workspace volumes.

### P1 -- Correctness

2. **Fix CEO self-assignment of research tasks.** The CEO prompt forbids direct research but the agent still self-assigned EVA-62 and EVA-66. Add a Paperclip-level constraint or triage-workflow hook that blocks CEO from assigning issues to itself when they match research/data/writing patterns.

3. **Fix stale context contamination.** CEO created 3 escalation issues (EVA-57, 58, 69) from old crash-loop comments. Either clear issue comments on fresh runs or add context freshness checks that distinguish historical errors from current state.

4. **Fix duplicate artifact publishing.** Agents publish the same document 2-5 times under different artifact_types. Add deduplication (content_hash check before write) or enforce a single canonical type per publish call.

### P2 -- Performance

5. **Investigate MiniMax timeout pattern.** 5 timeouts in 30 minutes, all following the same signature (successful start, silent hang). MiniMax API may hang on long tool-use chains. Consider adding a Pi-level keepalive or switching to DeepSeek for agents with complex tool flows.

6. **Reduce heartbeat noise during active work.** Bridge invocation lock serializes wakes, but heartbeat wakes still process "check inbox" prompts that waste 1-2 turns. Skip heartbeat processing when an active run is in progress.

7. **Fix duplicate write task creation.** CEO created 3 overlapping write tasks (EVA-56, 67, 68). Improve task deduplication in CEO decomposition prompt or add a pre-creation check for existing open issues with similar titles.

### P3 -- Data quality

8. **Fix S3 key structure.** Most artifacts use `no-run` and `default` placeholders. Ensure run_id and company_id propagate from bridge environment through to the artifact client on every write.

9. **Enforce artifact_type vocabulary.** Current types are free-text with no validation. Define a controlled vocabulary (e.g. research, brief, report, dataset, finding) and validate on write in the artifact service.

10. **Populate encouraged workproduct fields.** topic_tags, entities, and related_findings are all empty arrays. Add prompt engineering or post-processing to fill these, enabling cross-finding queries and entity graphs.
